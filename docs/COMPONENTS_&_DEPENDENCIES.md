This report outlines the Dependencies and Components Blueprint for your Zero-Knowledge Password Manager API.It focuses on building a "Best-in-Class" Rust stack that is 100% free for commercial use (MIT/Apache 2.0 licenses), high-performance, and strictly typed.Part 1: The Database & Cache Strategy (The License Issue)You raised a valid concern regarding Redis.The Situation: Redis changed its license to RSALv2/SSPLv1. While it is still free for internal use (you can use it for your app without paying), it is no longer strictly "Open Source" by OSI standards, which creates uncertainty for long-term projects.The Solution: Valkey.What it is: Valkey is the Linux Foundation’s fork of Redis. It is the new industry standard for an open-source, permissive alternative. It is 100% compatible with Redis.Does Valkey exist for Rust?The Server: Yes, you will run the Valkey Server (via Docker or Linux package) instead of Redis.The Client (Rust Crate): You do not need a special "Valkey Crate." Because Valkey maintains protocol compatibility, you use the standard, battle-tested Redis crates in Rust to talk to Valkey. They work seamlessly.Part 2: The Rust "Crate" Stack (Dependencies)Here is the optimal selection of libraries. All crates listed below use MIT or Apache 2.0 licenses, ensuring you have zero legal hurdles for commercial distribution or portfolio showcases.1. Core Web & Runtimetokio: The async runtime. It’s the engine of the application.Features needed: full (macros, rt-multi-thread, net, fs).axum: The Web Framework.Role: Handles routing, middleware, and request parsing. It is ergonomic and integrates deeply with Tokio.tower & tower-http: Middleware utilities.Role: Handles generic web features like Timeout, Tracing, CORS, and Compression.2. Data Persistence (Database & Cache)sqlx: The SQL Toolkit.Role: Connects to Postgres. Unlike .NET's EF Core, sqlx verifies your SQL queries at compile time. If you typo a column name, the code won't compile.Features: runtime-tokio, tls-rustls, postgres, uuid, chrono.deadpool-redis (using the redis crate): Connection Pooling for Valkey/Redis.Role: Manages a pool of connections to Valkey to ensure high concurrency without the overhead of opening a new connection for every request.3. Cryptography & Security (The "Heavy Lifting")argon2: Password Hashing & Key Derivation.Role: Deriving the Master Key from the user's password.chacha20poly1305: Authenticated Encryption.Role: The actual encryption/decryption of secrets and the "Canary". Faster than AES on mobile/non-hardware-accelerated CPUs.secrecy: Memory Protection.Role: Wraps sensitive data (Master Key) and automatically zeroes out the memory (writes 0000) when the variable goes out of scope. Critical for this project.rand: Random Number Generation.Role: Generating Salts and Refresh Tokens.4. Authentication & Tokensjsonwebtoken: JWT manipulation.Role: Creating and verifying the Access Tokens.sha2: Hashing algorithms.Role: Hashing the Refresh Tokens before storing them in Postgres (as per our blueprint).5. Concurrency & Utilitiesrayon: Data Parallelism.Role: Multithreading logic. When decrypting 50 secrets at once, rayon splits the work across all available CPU cores automatically.serde & serde_json: Serialization.Role: Converting Rust structs to JSON and back. The de-facto standard.thiserror: Library Error Handling.Role: Defining neat, strongly typed errors for your internal logic.tracing & tracing-subscriber: Observability.Role: Structured logging (JSON logs) to see exactly what is happening in the async threads.validator: Input Validation.Role: Validating email formats and password strength requirements on incoming requests.Part 3: Application Components & OrchestrationIn .NET, you think in "Controllers" and "Services". In Rust, we think in Modules and Traits. Here is the breakdown of the application units.1. The Orchestrator: AppStateIn Rust, we don't usually use a Dependency Injection Container (like Autofac). Instead, we create a single struct called AppState that holds our "Infrastructure Handles."Definition: A thread-safe struct shared across all API routes.Contains:db: A connection pool to Postgres (PgPool).cache: A connection pool to Valkey (deadpool::Pool).email_sender: An async client for sending verification emails.Orchestration: When the app starts, you initialize these pools once and pass them into the Axum Router. Every handler can then access the DB and Cache immediately.2. The Presentation Unit: handlersJob Definition: Parse HTTP requests, validate inputs, call the core logic, and map results to HTTP responses (200 OK, 401 Unauthorized).Example Units: auth_handler, secrets_handler.Key Behavior: Handlers must be "thin." They should not contain complex crypto logic; they just organize the data for the heavy lifters.3. The Logic Unit: core (The Domain)This is where the "Blocking Threads" live. This module contains the pure business logic, decoupled from the HTTP server.Job Definition: Perform the actual work: Decrypting, Hashing, Verifying Canaries.The "Crypto Service": A specific module responsible for all secrecy and argon2 operations.Why separate? Security auditability. If you need to upgrade encryption algorithms later, you change it in one file, not in 50 controllers.Orchestration: Handlers call tokio::task::spawn_blocking to run functions from this unit.4. The Data Unit: repositoryJob Definition: Strictly database queries. No business logic.Responsibility: "Find user by email", "Insert Secret", "Update Refresh Token".Interaction: The handlers or core logic call the repository methods using the db pool from AppState.Part 4: Missing Critical Matters (The "Unspoken" Needs)To make this a true production-grade system, we must add these three missing pillars to the plan:1. Configuration Management (config crate)The Problem: You have secrets (Database URL, JWT Signing Key, SMTP credentials). You cannot hardcode these.The Solution: Use the config crate. It allows a layered approach:Read Settings.toml (Default values).Read Settings.Production.toml (Environment specific).Read Environment Variables (Overwrites everything).Benefit: This is Cloud-Native compliant. You can deploy to Docker/Kubernetes and configure the app entirely via Environment Variables without recompiling.2. Database Migrations (sqlx-cli)The Problem: How do we create the users and secrets tables on a new server?The Solution: SQLx has a built-in migration system.You write SQL files: 2024011501_create_users.sql.The App runs these migrations automatically on startup (or via CI/CD).Benefit: Ensures the database schema is always in sync with the code version.3. The "Docker Compose" UnitThe Problem: You need to run Postgres, Valkey, and your App together for local development.The Solution: A docker-compose.yml file.Service 1: Postgres (postgres:16-alpine).Service 2: Valkey (Using valkey/valkey:9.0.1-alpine image).Service 3: Rust API (Your app).Orchestration: This file defines the network that allows your Rust app to talk to Valkey using the hostname valkey.Summary of the "Stack"LayerComponentImplementation / CrateLicenseAPIWeb FrameworkAxumMITApp LogicOrchestratorTokio (Async Runtime)MITDomainParallel ProcessingRayonMIT/ApacheSecurityEncryption/HashingArgon2 + ChaCha20Poly1305MIT/ApacheSecuritySensitive MemorySecrecyMIT/ApacheDataSQL ClientSQLx (Postgres)MIT/ApacheCacheCache ClientRedis (crate) + DeadpoolMITInfrastructureDatabase ServerPostgreSQLPostgreSQL (Free)InfrastructureCache ServerValkeyBSD (Free)This blueprint provides a completely free, open-source, and legally safe foundation for your application.
